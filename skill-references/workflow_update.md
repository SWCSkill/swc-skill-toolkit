# Workflow: обновление существующего скилла

Этот гид - для сценария UPDATE. Веди сотрудника по шагам, поведение должно соответствовать трёхуровневой модели вмешательства из SKILL.md (фоновая валидация / обычная работа / архитектурный аудит).

## Шаг 1. Какой скилл и какая задача

Спроси одним сообщением:
1. Какой скилл обновляем? (имя)
2. Какая задача? (примеры: «добавить новый файл», «поправить тон», «расширить раздел», «переадресовать в новый соседний скилл», «исправить опечатку»)

Если скилл - один из четвёрки, отлично, ты его знаешь. Если новый/чужой - тоже работаем, веди по этому же гиду.

## Шаг 2. Прочитай скилл и оцени состояние (фоновая валидация)

Перед любой работой - подтяни ключевые файлы скилла через curl:

```bash
# Корневой README, чтобы понять структуру
curl -s "https://raw.githubusercontent.com/SWCSkill/<repo>/main/README.md"

# SKILL.md.template - источник поведения
curl -s "https://raw.githubusercontent.com/SWCSkill/<repo>/main/skill-template/SKILL.md.template"

# VERSION
curl -s "https://raw.githubusercontent.com/SWCSkill/<repo>/main/VERSION"

# CHANGELOG.md - сверить, синхронизирован ли с VERSION
curl -s "https://raw.githubusercontent.com/SWCSkill/<repo>/main/CHANGELOG.md"

# Список файлов в skill-references/
curl -s "https://api.github.com/repos/SWCSkill/<repo>/contents/skill-references"
```

Прочитай и проверь:

1. **Валиден ли YAML frontmatter** в SKILL.md? Есть `name`, `description`?
2. **Совпадает ли VERSION с последней записью CHANGELOG**? (Если VERSION = 1.3.0, в CHANGELOG должна быть запись `[1.3.0]`.)
3. **Есть ли последний release**? `curl -sIL "https://github.com/SWCSkill/<repo>/releases/latest/download/<skill>.skill" | head -1` - должен дать `HTTP/2 302`.
4. **Есть ли GitHub Action**? `curl -s "https://raw.githubusercontent.com/SWCSkill/<repo>/main/.github/workflows/build-skill.yml"` - должен вернуть YAML.

Если что-то критично сломано (битый YAML, нет workflow, нет release) - **молча включи починку в текущую задачу** и одной строкой уведомь сотрудника. Не превращай это в обсуждение, если не задают вопрос.

## Шаг 3. Определи тип изменения

По задаче сотрудника пойми, что меняется:

| Тип изменения | Куда правки | Бамп версии |
| :--- | :--- | :--- |
| **Только контент** (новый файл в `articles/`, правка ИП, обновление FAQ) | Папка с контентом в репо | НЕ нужен |
| **Контент + мета** (новый файл + обновление INDEX.md в `skill-references/`) | Контент + skill-references | Нужен (PATCH или MINOR в зависимости от объёма) |
| **Поведение** (правка SKILL.md.template - тон, маршрутизация, новый раздел) | skill-template/ | Нужен (PATCH/MINOR/MAJOR) |
| **Инфраструктура** (правка scripts/build_skill.py или workflow) | scripts/ или .github/ | Нужен (обычно MINOR) |

Скажи сотруднику явно, что меняется и нужен ли бамп. Это часто непонятно неискушённым - помоги им понять.

## Шаг 4. Принципиальный ли это апдейт - архитектурный аудит при необходимости

Прежде чем приступать к самой правке, прокрути в голове:

**А) Решается ли задача в текущей архитектуре?**

Если сотрудник просит «динамически подгружать обновления» в скилле, у которого всё в SKILL.md - текущая архитектура этого не позволяет. **Нельзя промолчать.** Скажи:

> Чтобы это сработало, нужно сначала вынести <такую-то часть> в репо. Это перестройка - примерно <оценка времени>. Готов идти на это, или сначала разберёмся, что попроще?

**Б) Существенное архитектурное отклонение?**

При первом обращении к скиллу проверь - нет ли явных нарушений принципов из `architecture.md`:

- **Дублирование контента**: один факт лежит и в `references/` скилла, и в репо. Это рано или поздно даст разночтения.
- **Контент в SKILL.md**: длинные блоки данных (FAQ, описания продуктов, тексты документов) вшиты прямо в SKILL.md вместо репо.
- **Отсутствие сборки**: нет GitHub Action, релизы делаются вручную.
- **Hardcoded ссылки** на чужие репо/ID, которые должны быть в схеме переадресации.

Если нашёл такое - подними вопрос проактивно (это уровень 3 из SKILL.md):

> Я заметил, что у тебя <конкретное отклонение> - <короткое объяснение, почему это рискованно>. Хочешь, разберёмся вместе с твоей основной задачей? Это займёт <оценка> сверху. Или решим только текущую, а архитектуру - в другой раз.

Решение принимает сотрудник. **Не дави.**

**В) Косметические отличия от твоих ожиданий?**

Если у скилла другой стиль формулировок, иные имена в `references/`, иной порядок разделов - **не лезь**. У автора были причины. Это не отклонение от архитектуры, это его выбор.

## Шаг 5. Выполни правку

Дальше сценарии расходятся в зависимости от типа изменения.

### 5а. Только контент (новый файл, правка существующего)

Это самый простой случай. Сотрудник присылает контент в чат, ты:

1. Оформляешь его в `.md` по стилю существующих файлов в этой папке.
2. Подбираешь имя файла по конвенциям папки.
3. Если есть INDEX в `skill-references/` - готовишь к нему обновление.
4. Упаковываешь в архив (только новые/изменённые файлы) и отдаёшь сотруднику.

Версию **не бампаешь**. CHANGELOG **не трогаешь**. Скилл подхватит новый контент через curl автоматически.

Инструкция сотруднику:

> Распакуй и залей в репо `https://github.com/SWCSkill/<repo>` через Add file → Upload files. Файл(ы) нужно положить в папку <такую-то>. Commit message: `Update <папка>: add <тема>`. Action не сработает (мы только контент трогали), переустанавливать скилл не нужно. Через несколько минут новый контент появится в работе скилла.

### 5б. Поведение (правка SKILL.md.template)

1. Возьми текущий SKILL.md.template (через curl).
2. Внеси правки точечно через найди-и-замени, не переписывай файл целиком.
3. Бампни VERSION (PATCH/MINOR/MAJOR в зависимости от объёма - см. `conventions.md`).
4. Допиши запись в CHANGELOG (см. `changelog_style.md`).
5. Упакуй три-четыре файла в архив: `skill-template/SKILL.md.template`, `VERSION`, `CHANGELOG.md`, и - опционально - изменённые `skill-references/*.md`.

Инструкция сотруднику:

> Распакуй и залей разом через Add file → Upload files. Файлы заменят существующие. Commit message: `v<новая-версия>: <короткая суть>`. Action автоматически соберёт новый release `v<новая-версия>` в течение минуты.
>
> Проверь:
> 1. https://github.com/SWCSkill/<repo>/releases - должен появиться `v<новая-версия>`.
> 2. https://github.com/SWCSkill/<repo>/releases/latest/download/<skill>.skill - скачается обновлённый файл.
>
> У себя в Claude.ai: Settings → Skills → удалить старый <skill>, скачать новый по постоянной ссылке, Upload skill.

### 5в. Инфраструктура (build_skill.py, workflow)

Этот случай реже. Если правишь скрипт сборки или Action - **обязательно протестируй локально**:

```bash
cd <repo-clone>
python3 scripts/build_skill.py --output dist/test.skill
```

Должно собраться без ошибок, файл должен существовать, валидный ZIP, внутри `<skill-name>/SKILL.md` с подставленной версией.

После успешного локального теста - бампай VERSION (минимум MINOR), пакуй и отдавай сотруднику.

## Шаг 6. После заливки - проверь, что Action отработал

Если изменения требовали бампа версии:

```bash
sleep 60  # дай Action время

# Проверь HTTP-код
curl -sIL "https://github.com/SWCSkill/<repo>/releases/latest/download/<skill>.skill" | head -1
```

`HTTP/2 302` - всё хорошо, новая версия в Releases. `HTTP/2 404` - что-то пошло не так, скажи сотруднику посмотреть Actions tab в репо и прислать тебе логи.

## Шаг 7. Тестирование после обновления

Если правка касалась поведения (SKILL.md):

> Переустанови скилл и попробуй [конкретный сценарий, который должен работать после правки]. Скажи, как отвечает.

Если ответ корректный - готово. Если нет - возвращайся к шагу 4, разберитесь, что пошло не так. Возможно, нужен ещё PATCH.

## Что НЕ делать в сценарии UPDATE

- Не переписывай чужие референсы «по конвенциям». Уважай существующий стиль.
- Не предлагай рефакторинг при каждой задаче. Только если есть существенное отклонение или сотрудник сам спросил.
- Не делай архив с **всем содержимым** скилла. Только то, что меняется.
- Не забывай про CHANGELOG. Запись в нём - часть коммита, не отдельная задача.
- Не предлагай сотруднику «попробуй сам через git». Он работает в Claude.ai и веб-UI GitHub - этого достаточно.
