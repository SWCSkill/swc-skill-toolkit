# Архитектура корпоративных скиллов SWC

Этот файл описывает **почему** скиллы SWC устроены так, как устроены. Конкретные правила (как именно делать) - в `conventions.md`. Здесь - принципы.

Когда сотрудник спрашивает «почему такая структура» или «можно ли иначе» - объясняй из этого файла.

## Главный принцип: «контент в репо, поведение в скилле»

Скилл состоит из двух слоёв:

**Слой A. Содержимое** - конкретные факты, статьи, тексты, описания продуктов, термины. Это «что я знаю».

**Слой B. Поведение** - как искать, как отвечать, какой стиль, когда отказать, когда переадресовать. Это «как я работаю».

Слой A живёт **в репозитории на GitHub** и подтягивается скиллом через curl во время работы.

Слой B живёт **в SKILL.md** и referencing-файлах внутри пакета `.skill`. Он загружается в Claude.ai один раз при установке.

### Почему так

1. **Контент часто обновляется, поведение - редко.** ManaJet выпустил новую статью - надо положить файл в репо и всё работает. Это происходит раз в неделю. Поведение скилла перестраиваешь раз в полгода.

2. **Контент должен иметь один канонический источник.** Если факт лежит и в скилле, и в репо - они **обязательно** разойдутся через какое-то время, и скилл начнёт выдавать противоречия.

3. **Сотруднику не надо переустанавливать скилл при каждой правке контента.** Postоянная ссылка `releases/latest/download/<skill>.skill` обновляется только когда меняется поведение, что происходит редко.

### Что считается «контентом», а что «поведением»

| Это контент (живёт в репо) | Это поведение (живёт в скилле) |
| :--- | :--- |
| Статьи справки, ИП, FAQ | Маршрутизация запросов |
| Описания продуктов | Правила тона |
| Термины со значениями | Шаблоны URL и форматов |
| Регламенты, процедуры | Принципы переадресации |
| Примеры из жизни компании | Шаги диагностики |
| Списки сущностей (фонды, отделы, посты) | Список соседних скиллов |

Граница не всегда очевидна. Если сомневаешься - вопрос: **«если это завтра обновится, нужно ли пересобирать .skill?»**. Если да - значит, поведение, оставляй в скилле. Если нет - контент, в репо.

## Принцип №2: один источник правды по каждому факту

Этот принцип следует из первого, но достаточно важен, чтобы выделить отдельно.

**Никогда** не дублируй один и тот же факт в `references/` скилла и в репо. Это всегда заканчивается разночтениями. Если факт нужен и тут и там - оставляй его **только в репо**, а в скилле оставь ссылку «читай в репо такой-то файл».

Эталонный пример - `swc-manajet-expert`. Там в скилле только мета (карта статей INDEX, словарь терминов, методические практики), а сами 221 статья справки - **только в репо**. Двух мест, где можно править содержание статьи 165 «Формульная статистика», не существует.

Контр-пример - `swc-assistant`, в его текущей версии (v1.0.2) описания продуктов SWC лежат в `references/` скилла, но при этом FAQ про эти же продукты - в репо. Это **архитектурное отклонение**, которое со временем приведёт к разночтениям. Если заметишь это в работе с swc-assistant - уровень 3 (см. SKILL.md): подними вопрос с автором.

## Принцип №3: слой меты для навигации

Внутри `references/` скилла должны быть навигационные файлы - INDEX, словари, карты, маршрутизация. Они помогают скиллу быстро решить, **где** в репо искать ответ, не подгружая всё подряд.

Пример из ManaJet Expert:
- `references/INDEX.md` - карта 221 статьи по разделам, без содержимого
- `references/swc_terminology.md` - словарь соответствий ManaJet ↔ SWC
- `references/swc_practice.md` - методические надстройки

Эти файлы **локальны** (читаются мгновенно через `view`), а контент по найденному ID - подтягивается curl-ом. Так скилл работает быстрее и не тратит контекст на ненужные данные.

Когда проектируешь новый скилл - подумай, **какие меты нужны**. Без INDEX скилл не сможет эффективно ориентироваться в большом корпусе. Если корпус маленький (5-10 файлов) - INDEX можно не делать.

## Принцип №4: слоты под рост заранее

При проектировании скилла - заранее закладывай категории контента, которые могут расти. Это разделение помогает потом **наполнять без изменений в SKILL.md**.

Например, у `swc-orgpolicies-guru` (планируется) сразу должны быть слоты:
- `instructional_letters/` - инструктивные письма
- `hat_writeups/` - шляпы / настольные руководства
- `policies/` - регламенты
- `templates/` - шаблоны документов

Каждая папка может расти независимо. Добавили 30-е ИП - просто положили файл, скилл его подхватит. Не надо ничего переписывать.

**Антипаттерн**: одна большая папка `documents/` со всеми типами вместе. Тогда при появлении нового жанра придётся править SKILL.md.

## Принцип №5: гарантированный канал распространения

Каждый скилл SWC имеет **постоянную ссылку** на актуальную версию:

```
https://github.com/SWCSkill/<repo>/releases/latest/download/<skill-name>.skill
```

Эта ссылка не меняется никогда. Сотрудник по ней получает всегда последнюю версию. Внутри организации этот URL - **единственный канал** для распространения скилла. Никаких «отправь файл в чате», «сохрани у себя в Drive» - только постоянная ссылка.

Это работает благодаря GitHub Action, который при каждом push изменений автоматически собирает `.skill` и делает release.

## Принцип №6: переадресация, не подмена

Скиллы экосистемы знают друг о друге и **переадресуют** запросы вне своей зоны. Никогда не пытаются ответить за чужой скилл. Список соседей в SKILL.md, проверка готовности соседа через curl на `releases/latest/download`.

Если сосед готов (HTTP 302) - переадресуй на установку. Если в разработке (HTTP 404) - честно скажи и предложи альтернативу. Не выдумывай ответы, которые должен давать другой скилл.

## Что из этого следует для нового скилла

Когда помогаешь сотруднику создавать новый скилл - в этом порядке:

1. **Определить зону**: что покрывает скилл, что не покрывает (где границы с соседями).
2. **Определить корпус**: какие данные войдут, как они структурированы.
3. **Провести границу контент/поведение**: что в репо, что в скилле.
4. **Спроектировать слоты**: какие папки в репо, какие меты в скилле.
5. **Спроектировать переадресации**: к каким соседям отправлять и в каких случаях.
6. **Сборка инфраструктуры**: VERSION, CHANGELOG, GitHub Action, INSTALL.md.

Шаги 1-5 - совместная работа с автором. Шаг 6 - механика, делается через скелет в `swc-skill-toolkit/skeleton/`.
