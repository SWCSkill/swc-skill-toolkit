# Конвенции корпоративных скиллов SWC

Конкретные правила оформления. Принципы, лежащие в основе - в `architecture.md`. Здесь только «как именно».

## Нейминг

### Имена скиллов и репо

Скиллы лежат в организации **SWCSkill** на GitHub. Имена скиллов и репо - **kebab-case** (с дефисами).

| Префикс / суффикс | Что означает | Примеры |
| :--- | :--- | :--- |
| `swc-*` | Скилл, относящийся к SWC как компании | `swc-assistant`, `swc-skill-toolkit` |
| `*-expert` | Знаток внешней системы или предметной области | `swc-manajet-expert` |
| `*-guru` | Знаток внутренних правил и норм компании | `swc-orgpolicies-guru` |
| `*-helper` | Вспомогательный инструмент для конкретных задач | `manajet-api-helper` |
| `*-toolkit` | Инструментарий для разработчиков скиллов | `swc-skill-toolkit` |
| `*-assistant` | Универсальный помощник по теме | `swc-assistant` |

При создании нового скилла - имя выбирается осознанно. Если зона = внешняя система → `<system>-expert`. Если зона = внутренние правила → `swc-<area>-guru`. И т.д.

### Имя репо vs имя скилла

Обычно совпадают, но есть исторические исключения:

| Скилл | Репо |
| :--- | :--- |
| `swc-assistant` | `swc-knowledge-base` (исторически) |
| `swc-manajet-expert` | `manajet-knowledge-base` (исторически) |
| `swc-orgpolicies-guru` | `swc-orgpolicies-guru` (новое) |
| `manajet-api-helper` | `manajet-api-helper` (новое) |
| `swc-skill-toolkit` | `swc-skill-toolkit` (новое) |

**Для новых скиллов имя репо = имя скилла.** Без `-knowledge-base` суффикса. Это упрощает понимание.

### Имя файла .skill

Файл, который сотрудник скачивает и устанавливает в Claude.ai, называется так же, как скилл, плюс `.skill`:

```
swc-manajet-expert.skill
swc-orgpolicies-guru.skill
manajet-api-helper.skill
```

В release лежат **два** файла - стабильное имя (`<skill-name>.skill`) и версионированное (`<skill-name>-v1.3.0.skill`). Ссылка `releases/latest/download/<skill-name>.skill` ведёт на стабильное имя.

## Структура репо

Эталон - `manajet-knowledge-base` (репо для swc-manajet-expert).

```
<repo>/
├── articles/                        ← живой контент (или другая папка под жанр)
│   ├── 1.md
│   ├── 2.md
│   └── ...
├── INDEX.md                         ← публичный индекс контента (опционально)
│
├── skill-template/
│   └── SKILL.md.template            ← шаблон с {{VERSION}} и {{BUILD_DATE}}
│
├── skill-references/                ← локальные референсы скилла
│   ├── INDEX.md                     ← навигация для скилла (опционально)
│   ├── <other_references>.md
│
├── scripts/
│   ├── build_skill.py               ← сборка .skill из шаблона + skill-references
│   └── <other_scripts>.py           ← опционально (build_corpus.py и т.п.)
│
├── .github/workflows/
│   └── build-skill.yml              ← автосборка release при push
│
├── VERSION                          ← текущая версия в формате `1.3.0\n`
├── CHANGELOG.md                     ← история версий
├── INSTALL.md                       ← инструкция установки для сотрудников
└── README.md                        ← описание репо для разработчиков
```

Имена папок `articles/`, `skill-template/`, `skill-references/`, `scripts/`, `.github/workflows/` - **обязательны** и стандартны. Имя папки с контентом (`articles/`) может отличаться - важно, чтобы оно было одно (а не «documents и files и data», размазанные данные).

## Версионирование (semver для скиллов)

Версия в файле `VERSION` (одна строка, без префикса `v`):

```
1.3.0
```

Семвер интерпретируется так:

- **MAJOR** (1.x.x → 2.0.0) - перестройка архитектуры, разрыв обратной совместимости, серьёзное изменение поведения. Сотрудник должен переустановить и проверить, что его привычные сценарии работают.
- **MINOR** (1.3.x → 1.4.0) - значительное новое поведение, новый раздел SKILL.md, новый референс. Обратная совместимость сохраняется, но появляются новые возможности.
- **PATCH** (1.3.0 → 1.3.1) - точечная правка тона, формулировок, мелких ссылок, опечаток. Поведение принципиально не меняется.

Как **не** работает семвер для скиллов:
- Не считаем «количество строк изменений» - считаем влияние на сотрудника.
- Не делаем 0.x версии (это не библиотека, это инструмент - стартуем с 1.0.0).
- Не делаем pre-release метки (`-alpha`, `-rc`) - либо есть release, либо нет.

### Когда **не** надо бамплать версию

- Изменили только контент в репо (статью, ИП, FAQ) - **не пересобираем**, не бамплим. Контент скилл подхватит сам.
- Изменили только README или INSTALL - тоже не повод для нового release.
- Опечатка в CHANGELOG старой версии - правишь и не бамплишь.

### Когда **надо** бамплать

- Любое изменение `skill-template/SKILL.md.template` или файлов в `skill-references/`.
- Изменение `scripts/build_skill.py` (если меняется логика сборки).
- Изменение `.github/workflows/build-skill.yml`.

GitHub Action настроен так, что push в эти места автоматически собирает новый release - но только если бампнут VERSION. Если поправил SKILL.md, но забыл бампнуть версию - Action попытается выпустить ту же версию заново, что иногда срабатывает, иногда нет. Поэтому **бамп VERSION = обязательная часть коммита с правкой поведения**.

## Файл VERSION

Простой текстовый файл, одна строка, формат `MAJOR.MINOR.PATCH\n`:

```
1.3.0
```

Никаких комментариев, никаких пробелов. Чистая версия с переводом строки в конце.

## CHANGELOG

См. отдельный файл `changelog_style.md`.

## YAML frontmatter в SKILL.md

Каждый SKILL.md начинается с YAML-блока:

```yaml
---
name: skill-name-kebab-case
description: >-
  Описание для активации скилла. Должно содержать триггеры
  (фразы, при которых Claude должен активировать скилл) и
  отграничения (когда не использовать).
---
```

`description` - **многострочный**, с явным описанием когда использовать и когда не использовать. Анти-триггеры (когда переадресовать к соседу) явно перечислены.

## Раздел «Соседние скиллы» в SKILL.md

Обязательный раздел в каждом скилле. Содержит:

1. Таблицу всех соседних скиллов с указанием зоны и репо.
2. Инструкцию по проверке готовности соседа через curl.
3. Шаблоны переадресации для случаев «сосед готов» и «сосед в разработке».

См. эталон - SKILL.md от swc-manajet-expert.

## Раздел диагностики и helping setup

Каждый скилл, использующий curl к raw.githubusercontent.com, должен включать раздел «Помощь сотруднику с настройкой доступа» с инструкцией по добавлению `raw.githubusercontent.com` в Settings → Code execution → Additional allowed domains.

Без этого раздела сотрудник встретит ошибку и не поймёт, что делать.

## GitHub Action

Универсальный workflow `.github/workflows/build-skill.yml` идентичен между всеми скиллами SWC. Берётся из скелета без изменений.

Триггеры:
- `push` в `main` при изменении `skill-template/`, `skill-references/`, `scripts/build_skill.py`, `VERSION`, `.github/workflows/build-skill.yml`.
- `workflow_dispatch` - ручной запуск через UI.

Что делает:
1. Читает VERSION.
2. Запускает `scripts/build_skill.py` с подстановкой версии и даты.
3. Создаёт релиз `v<version>` (или обновляет существующий).
4. Прикрепляет два файла: `<skill>.skill` (стабильное имя) и `<skill>-v<version>.skill`.

## Имена коммитов

Минимально:
- При обновлении контента: `Update <что>` (например, `Update articles: add 222-th statistic article`).
- При обновлении поведения: `v<version>: <короткая суть>` (например, `v1.4.0: add Neighboring Skills section`).
- При первичной заливке: `Initial commit` или `Initial corpus`.

Не обязательно использовать Conventional Commits, но be specific. Сообщения вида «update» или «fix» - не годятся.
