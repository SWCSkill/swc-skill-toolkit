# Workflow: создание нового скилла

Этот гид - для сценария CREATE (новый скилл с нуля). Веди сотрудника по шагам по очереди, не пропуская. Никаких длинных монологов: один шаг - один обмен с сотрудником, потом следующий.

В конце сценария - сотрудник получает zip-архив со всеми файлами для нового репо и инструкцию по заливке.

## Шаг 1. Кратко выясни задачу

Прежде чем планировать архитектуру, тебе нужно три вещи. Спроси одним сообщением, без лишнего:

1. **Зачем нужен скилл?** Какую задачу должен решать. Кому помогать.
2. **Какой контент пойдёт внутрь?** ИП, FAQ, инструкции, описания продуктов, статьи - что-то одно или смесь?
3. **Какое имя в SWCSkill?** Если сотрудник не уверен - помоги выбрать по конвенциям (`conventions.md`, раздел «Нейминг»).

Не задавай 10 вопросов за раз. Этих трёх достаточно для старта - дальше будут уточняющие вопросы по ходу.

## Шаг 2. Проверь, что зона не пересекается с существующими

После ответа на первый вопрос - сверься с картой четвёрки:

| Скилл | Зона |
| :--- | :--- |
| swc-assistant | Продукты, бренд, партнёрка SWC |
| swc-manajet-expert | Работа в системе ManaJet |
| swc-orgpolicies-guru | Орг.политика SWC, ИП, шляпы |
| manajet-api-helper | API ManaJet |

Если новая идея **сильно пересекается** с существующим - не торопись создавать новый скилл. Спроси: «Тебе действительно нужен отдельный скилл, или это можно добавить в существующий <такой-то>?»

Расширить существующий обычно проще. Новый скилл оправдан, когда:
- зона ответственности явно своя и не пересекается;
- аудитория иная (например, разработчики vs обычные сотрудники);
- контент нельзя положить в существующий без нарушения границ.

Если сотрудник настаивает на новом - продолжай. Решение за ним.

## Шаг 3. Спроектируй контент: что лежит в репо

Здесь ты помогаешь сотруднику решить, **какой будет структура репо**. Это самый важный шаг - от него зависит всё остальное.

Веди обсуждение примерно так:

> Окей, скилл будет про <тема>. Давай прикинем структуру. Я буду спрашивать, ты отвечать.
>
> Сколько примерно у тебя источников? Файлов / документов / разделов?
> Они однотипные (например, 30 ИП все одного формата) или разные жанры?

После ответа - предложи **папки в репо**. Принципы из `architecture.md`:

- Каждый жанр = своя папка. Не смешивать «documents/» с разнородным контентом.
- Имена папок понятные: `instructional_letters/`, `hat_writeups/`, `policies/`, `articles/`, `faq/`, `templates/`.
- Если всё однородное - одна папка достаточно (как `articles/` у swc-manajet-expert).

Покажи сотруднику предлагаемую структуру:

```
<repo>/
├── <папка-1>/      ← <что туда>
├── <папка-2>/      ← <что туда>
└── ...
```

Спроси: «Подходит? Поправить что-то?»

## Шаг 4. Спроектируй мету: что лежит в скилле

Решите вместе, **какие референсы нужны внутри скилла** в `skill-references/`.

Возможные:

- **`INDEX.md`** - карта контента в репо (если корпус большой - >20 файлов). Без содержимого, только перечисление с ID и темами.
- **`<terminology>.md`** - словарь терминов, если в зоне скилла есть свой жаргон.
- **`<practice>.md` / `<methodology>.md`** - методические надстройки, если есть знания о «как у нас принято делать», которые не выносятся в репо.

**Ключевое правило**: в `skill-references/` НЕ дублируется контент из репо. Только мета.

Спроси: «Какие меты нужны? Я предлагаю N штук такие-то. Согласен?»

## Шаг 5. Спроектируй маршрутизацию запросов

Теперь решите, **какие типы вопросов скилл обрабатывает**, а какие переадресует.

Задай вопросы:
- Какие типы запросов вы ожидаете? Опиши 3-5 типичных.
- На какие из них скилл отвечает сам?
- Какие переадресует к соседям (по карте четвёрки)?

После ответов - предложи таблицу маршрутизации (как в SKILL.md от swc-manajet-expert, шаг «Определить тип вопроса»).

Также продумайте, **как должен выглядеть ответ**: с ссылками на конкретные источники, со скриншотами (если контент в репо со скриншотами), в каком тоне.

## Шаг 6. Соседи и переадресации

Спросите сотрудника:
- В какие из четырёх скиллов будет переадресовывать новый скилл?
- К какому источнику отправлять, когда сосед в разработке?

Сформулируйте раздел «Соседние скиллы» по эталону из swc-manajet-expert.

## Шаг 7. Стартовый контент - что важно положить в репо первой версии

Сотрудник может начать с малого. Минимальный набор для первого release - 3-5 файлов в каждой папке. Дальше можно наполнять без переустановки скилла.

Спроси: «Какие 3-5 файлов положим в первый коммит? Можешь скинуть прямо в чат - я подготовлю их для репо».

Если у сотрудника есть **большой корпус** в форматах (PDF, Word, JSON) - предложи отдельный шаг: преобразование в Markdown. Если это сложно - предупреди, что для конвертации больших форматов лучше делать отдельный скрипт-конвертер (как `build_corpus.py` для ManaJet).

## Шаг 8. Сборка скелета

После того как все решения приняты - **собери архив** для сотрудника.

Подтяни заготовки из скелета:
```bash
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/SKILL.md.template.example
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/build_skill.py
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/build-skill.yml
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/README.md.example
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/INSTALL.md.example
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/CHANGELOG.md.example
curl -s https://raw.githubusercontent.com/SWCSkill/swc-skill-toolkit/main/skeleton/VERSION.example
```

Дальше:

1. **Адаптируй SKILL.md.template** под кейс сотрудника:
   - Заполни YAML frontmatter (name, description с триггерами).
   - Заполни роль и архитектуру.
   - Подставь маршрутизацию из шагов 5-6.
   - Подставь раздел «Соседние скиллы».
   - Версия: `{{VERSION}}` (не подставляй конкретную - это сделает Action).

2. **Создай файлы skill-references/** под решения шага 4.

3. **Скопируй без изменений** в репо:
   - `scripts/build_skill.py`
   - `.github/workflows/build-skill.yml`
   - `VERSION` со значением `1.0.0`
   - `CHANGELOG.md` с первой записью под `[1.0.0]`
   - `INSTALL.md` с подставленным именем скилла и репо
   - `README.md` адаптированный под скилл

4. **Положи стартовые файлы контента** (из шага 7) в нужные папки.

5. **Упакуй всё в zip** через bash + zip, отдай через `present_files`.

## Шаг 9. Инструкция по заливке

Сотрудник получает архив. Чтобы он не потерялся - дай ему **точный план**:

> 1. Создай новый репо в `https://github.com/SWCSkill/`. Имя: `<skill-name>`. Public, без README, без .gitignore, без license.
> 2. Распакуй архив локально.
> 3. На странице нового репо: Add file → Upload files → перетащи всё содержимое архива.
> 4. **ВАЖНО**: папка `.github/workflows/` через UI заливается сложнее. Если она не подхватилась - залей её отдельно: Add file → Create new file → имя файла `.github/workflows/build-skill.yml` → вставь содержимое (я отдам отдельно).
> 5. Commit message: `Initial commit - <skill-name> v1.0.0`.
> 6. Жди ~1 минуту - GitHub Action соберёт первый release.
> 7. Проверь: `https://github.com/SWCSkill/<skill-name>/releases/latest/download/<skill-name>.skill` - должен начаться скачивание.

Если на каком-то шаге проблема - сотрудник возвращается в чат с вопросом, ты помогаешь.

## Шаг 10. Установка у себя и тест

После того как Action успешно собрал release - сотрудник:

1. Скачивает скилл по постоянной ссылке.
2. Settings → Capabilities → Skills → Upload skill.
3. Settings → Capabilities → Code execution → Additional allowed domains → добавляет `raw.githubusercontent.com`, если ещё не добавлен.
4. В новом чате задаёт 3-5 тестовых вопросов из тех типов, которые планировались в шаге 5.

Если что-то отвечает не так - возвращается в чат, обсуждаете, готовите v1.0.1 или v1.1.0 (см. `workflow_update.md`).

## Что НЕ делать в сценарии CREATE

- Не пиши длинный SKILL.md за один присест без обсуждения.
- Не предлагай 5 архитектурных альтернатив - предложи **одну** и спроси «подходит?». Если нет - предложи вторую. Не больше.
- Не добавляй в первую версию «всё что может пригодиться». Минимум жизнеспособного. Расширять - в `workflow_update.md`.
- Не пиши контент за сотрудника. Ты помогаешь со структурой и оформлением, контент - его работа.
- Не привязывайся к одному решению. Если в процессе что-то не сходится - вернитесь к шагу 3 или 4 и пересоберитесь.
